## 2024-04-11 - Fixed Main Player Missing in Elo Mode
**Mode:** Medic
**Learning:** The 'Elo -> Win Rate' mode excluded the main player from the resulting dataset (`allResults`) and inadvertently hid the main player's inputs via CSS. It was critical to understand that `performCalculation` handles both modes, and in the `else` branch (Elo mode), the main player (`elements.mainElo.value`) must be manually added to the list of `allResults` just like in Win mode. Also important to recall to not use tools or package managers directly in the repo root to keep the standalone HTML environment clean.
**Action:** When working on standalone HTML files without build systems, always isolate test scripts in a temporary `verification/` folder and explicitly `rm -rf verification` before submitting. Additionally, always trace the lifecycle of variables between UI states and calculation inputs.
## 2024-04-18 - Validation Regression on Refactor
**Mode:** Razor
**Learning:** During refactoring of repeated boolean conditions into semantic getters (`isWinInput`, `isEloInput`), I inadvertently changed an edge-case interaction logic flow. I grouped an explicit array-index check (`this.isInvalid(this.competitors[0])`) into a generic `.some(c => this.isInvalid(c))` across the whole array. Because `this.competitors` tracks the underlying data independently of the UI mode's visibility, validating the entire array caused false positives for elements that were hidden or theoretically inactive in the specific sub-mode.
**Action:** When performing `Razor` refactors, strictly preserve the scope of the operations. If the original code singles out an array index (like `[0]`) based on the mode, do not broaden the scope to the whole array, even if the condition reads more clearly.
