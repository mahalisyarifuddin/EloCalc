## 2024-04-11 - Fixed Main Player Missing in Elo Mode
**Mode:** Medic
**Learning:** The 'Elo -> Win Rate' mode excluded the main player from the resulting dataset (`allResults`) and inadvertently hid the main player's inputs via CSS. It was critical to understand that `performCalculation` handles both modes, and in the `else` branch (Elo mode), the main player (`elements.mainElo.value`) must be manually added to the list of `allResults` just like in Win mode. Also important to recall to not use tools or package managers directly in the repo root to keep the standalone HTML environment clean.
**Action:** When working on standalone HTML files without build systems, always isolate test scripts in a temporary `verification/` folder and explicitly `rm -rf verification` before submitting. Additionally, always trace the lifecycle of variables between UI states and calculation inputs.
## 2024-04-18 - Validation Regression on Refactor
**Mode:** Razor
**Learning:** During refactoring of repeated boolean conditions into semantic getters (`isWinInput`, `isEloInput`), I inadvertently changed an edge-case interaction logic flow. I grouped an explicit array-index check (`this.isInvalid(this.competitors[0])`) into a generic `.some(c => this.isInvalid(c))` across the whole array. Because `this.competitors` tracks the underlying data independently of the UI mode's visibility, validating the entire array caused false positives for elements that were hidden or theoretically inactive in the specific sub-mode.
**Action:** When performing `Razor` refactors, strictly preserve the scope of the operations. If the original code singles out an array index (like `[0]`) based on the mode, do not broaden the scope to the whole array, even if the condition reads more clearly.
## 2024-04-25 - Fix focus navigation on Enter key for Elo inputs
**Mode:** Palette
**Learning:** Keyboard navigation for custom form fields needs to accurately track the visibility state of inputs. When implementing features like 'Dual Item Mode' which reuses elements and controls visibility via CSS attributes, generic DOM sibling traversal or checking just the base `this.mode` causes focus traps or skips. Additionally, root level elements like `#mainName` and `#mainElo` must be explicitly sequenced in keydown handlers.
**Action:** Always use established semantic getters (e.g. `this.isEloInput`) rather than raw string checks (`this.mode === 'elo'`) when determining focus targets. When adding new input fields (`#mainElo`), ensure they are added to custom navigation sequences (like `Enter` key handling) instead of being bypassed.
## 2024-05-02 - Fix Data Loss on Mode Switch
**Mode:** Medic
**Learning:** Destructive operations on underlying application state (like array slicing `this.competitors.slice(0, 1)`) to achieve visual UI constraints causes permanent data loss if the user switches out of that mode. When managing UI visibility based on application modes, CSS should be used to hide elements rather than mutating the core data structure. Also, it is critical to ensure that keyboard navigation logic (`focusNext`) does not attempt to focus elements that have been hidden via CSS (`display: none`), as this will create a focus trap or skip to unintended elements.
**Action:** When implementing different views or modes that require fewer data points than others, hide the extra elements visually using CSS selectors instead of deleting the data. Always cross-check keyboard event handlers to ensure they respect the visibility state of elements dictated by CSS when iterating or finding the "next" element.
